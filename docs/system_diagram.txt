                                  TAGCLEAN — END-TO-END SYSTEM
                              (Bengali EC/NID FAQ corpus cleaner)
                              [deterministic-only branch — E5 + heuristics; no LLM]


INPUTS
==============================================================================================
  question_tag.csv     -- ~79k rows × 1394 tags. Columns: question, tag (UTF-8, BOM-tolerant).
  tag_answer.json      -- { tag: canonical_answer }. Used as merge-safety gate in Stage 3.
  config.yaml          -- e5_instruction, thresholds, top_n, ...

MODELS
==============================================================================================
  E5-multilingual-large-instruct (1024d) -- single source of geometry.
                                            Same model production uses at inference.
                                            Composite weight: 90% (own 55 + margin 35).

  No LLM calls anywhere on this branch.


CORPUS-WIDE BOOTSTRAP  (run once; cached for all family runs)
==============================================================================================
  Stage 0  intake / Bengali normalize / dedup
           -> stage0/intake.parquet, cross_tag_duplicates.jsonl
  Stage 1  E5 embeddings + FAISS IndexFlatIP over the FULL corpus
           -> stage1/{emb_e5, emb_e5_query}.npy + faiss_e5.idx
  Stage 2  per-tag profile: outlier-trimmed centroid, medoid, central exemplars
           -> stage2/{tag_profile.parquet, tag_centroids_e5.npy, tag_index.json}


SCALING DISCOVERY  (one-shot; emits a hand-editable manifest)
==============================================================================================
                       stage2/tag_centroids_e5.npy + tag_index.json
                                         |
                                         v
  +----------------------------------------------------------------------------------+
  | tagclean discover                                                                 |
  |   --centroids-from <run>                                                          |
  |   --threshold 0.88  --pair-threshold 0.86  --top-k 8  --max-family-size 4         |
  |   [--exclude-tag-pattern '_followup_[a-d]$']    <- skip dialog-turn artifacts     |
  |   [--production-tags FILE]                      <- optional allowlist             |
  |                                                                                  |
  | Algorithm (E5-only seed-centric reciprocal-NN ego-family expansion):              |
  |   1. cos_e5 per pair; top-K neighbors per tag.                                    |
  |   2. Edge kept iff reciprocal AND >= threshold.                                   |
  |   3. Per seed: ego-family = seed + reciprocal nbrs, capped at max_family_size,    |
  |      requiring pairwise min_sim >= pair_threshold with every existing member.     |
  |   4. Score candidate by (min_edge_sim, avg_edge_sim, size).                       |
  |   5. Greedy descending selection; mark members covered.                           |
  |   6. Uncovered tags become singletons (full coverage guaranteed).                 |
  |                                                                                  |
  |   No transitive closure -> no mega-components at corpus scale.                    |
  +----------------------------------------------------------------------------------+
                                         |
                                         v
                                  families.yaml
                            (deterministic family_id from
                             sha256 of sorted tag set)


PER-FAMILY PIPELINE  (run-families dispatches one stage8 invocation per approved family)
==============================================================================================
  +--------------------------------------------------------------------------+
  | tagclean run-families --manifest families.yaml                            |
  |   [--include-singletons]                                                  |
  |   [--no-skip-completed]                                                   |
  |                                                                          |
  |   For each approved family:                                              |
  |     1. Symlink stage0/1/2 from `centroids_run_id` -> family run dir       |
  |        (stages 0-2 are corpus-wide, identical across families;            |
  |         symlinking saves ~30-60 min Mac MPS embedding per family.)        |
  |     2. Run stage8 with --target-tags <family target_tags> --run-id <fid>  |
  |     3. Skip if <fid>/stage8/cleaning_report.json already exists.          |
  +--------------------------------------------------------------------------+
                                         |
                                         v
  Stage 3  deterministic merge map (NO LLM)
           target_tags-aware: when family >= 2 tags, cluster IS target_tags.
           Per-pair deterministic check: token overlap on tag names + answer-token
           overlap on tag_answer.json (when provided) as merge-safety gate.
           -> stage3/{tag_merge_map.csv, merge_candidates.jsonl}
  Stage 4  deterministic ranker (E5-only)
           composite_score = 0.55*cos_e5 + 0.35*margin_e5
                           - 0.05*near_dup - 0.10*xtag_dup - 0.10*artifact
           -> stage4/row_features.parquet
  Stage 5  (DELETED — no LLM Stage QA reviewer on this branch)
  Stage 6  selection + MMR (lambda=0.7 in E5 space) -> top-N=40 per tag
           -> stage6/{question_tag.cleaned.csv, .top40.csv, jettisoned_rows.csv}
  Stage 7  (DELETED — long ago)
  Stage 8  leave-one-out validation (E5-only retrieval)
           -> stage8/cleaning_report.json


CROSS-FAMILY PRODUCTION COMPOSITION
==============================================================================================
  fam_xxxx/stage6/top40.csv  ---+
  fam_yyyy/stage6/top40.csv  ---+--> tagclean compose --manifest families.yaml
  fam_zzzz/stage6/top40.csv  ---+    [--compose-source top40 | cleaned]    --> composed.csv
                ...                  (concat + dedup + deterministic sort)
                                          |
                                          v
                                    tagclean stage9 --e5-audit-input <composed.csv>
                                    (E5 LOO top-K audit; drops top-1 misses)
                                          |
                                          v
                                    PRODUCTION CSV
                                    runs/<run_id>/stage9/production_filtered.csv

  WHY compose from top40 (or cleaned), not per-family stage9 production_filtered:
    A row that fails LOO inside its own family can be safely separable in the global
    production union -- its in-family rival is no longer the closest peer. Filter once
    globally, after composition. Per-family stage9 stays useful for diagnostics.


STAGE 9 INTERNALS  (E5-only production-risk audit)
==============================================================================================
  Input CSV -> Bengali normalize -> E5 prefix -> encode (passages = queries)
            -> chunked vecs @ vecs.T   (numpy, default chunk=4096; no FAISS to
               avoid Py3.14/Mac segfault when both libs load in same process)
            -> diag = -inf -> argpartition top-K + sort
            -> per row: top1_neighbor_tag, top1_correct, own_share_top_K, neighbor_dist
            -> e5_neighbor_audit.csv, production_filtered.csv,
               e5_dropped.csv, audit_report.json

  Default: drop rows with top-1 LOO mismatch.  --no-e5-drop = report-only mode.
  Stage 9 NEVER chains stage0-8 (auto-chaining clobbered cleaned.csv when geometry
  config differed). Run stage8 (or run-families) first; stage9 only audits what's there.

==============================================================================================

   data/question_tag.csv  ---->  [stage8 once on a 3-tag family --> stage0/1/2 cached]
                                                              |
                                                              v
                                  +---------------------------+
                                  |
                                  v
                          tagclean discover  --centroids-from <run>
                          --exclude-tag-pattern '_followup_[a-d]$'
                                          |

                                          |

                       (optional) human edits status: approved/rejected
                                          |

                          tagclean run-families  --manifest families.yaml
                          (one stage8 per approved family;
                           stage0/1/2 symlinked from centroids run;
                           --skip-completed default; resumable)
                                          |
                                          v
                          tagclean compose  --manifest families.yaml
                          --compose-source top40 --out PATH
                                          |
                                          v
                          tagclean stage9  --e5-audit-input PATH
                          --run-id production
                                          |
                                          v
                  runs/production/stage9/production_filtered.csv
                          (final E5-ready clean data)


NOTE ON VALIDATION HISTORY
==============================================================================================
  Earlier validation numbers (account_locked / name_correction / otp, top-1 LOO 0.961,
  0/14 cross-family drops) were measured against pipelines that included LLM-assisted
  stages (GPT or Claude). The deterministic-only branch retains the geometry-driven
  stages (0/1/2/4/6/8/9). The expected accuracy delta against the LLM-assisted
  baselines is small (sub-1% to low-single-digit top-1 retrieval) but should be
  re-measured against your dataset before shipping.
